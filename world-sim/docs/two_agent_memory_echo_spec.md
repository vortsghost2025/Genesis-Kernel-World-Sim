# 10Z – Two‑Agent Memory Echo Spec

## 1. Purpose
Define the first remembered‑heartbeat contract (Phase 10Z) that follows the 10Y single‑tick ingress harness.  It establishes the smallest two‑agent, two‑tick memory‑echo contract that can be exercised with existing mapper, verifier, ledger and exporter components.

## 2. Non‑goals
* No daemon, scheduler, provider, or autonomy‑loop code.
* No sanitizer implementation, artifact mutation, or runtime wiring.
* No economy, factions, population, or world‑scaling logic.
* No machine-specific identifiers, connection details, credential material, non-public notes, or transcript-language handling.
* Do not claim Adam or Eve are live agents.
* Do not claim memory is truth.
* Do not introduce a new `claim_scope`.


## 3. Two‑agent memory echo definition
A **memory echo** is a two‑tick, two‑agent sequence:
1. **Tick 1** – Adam records an *observed* event.
2. **Tick 2** – Eve produces a dependent event that references the Tick 1 record
   without promoting its truth status.  The dependent event may travel via a
   *speech* lane today and a future *memory* lane later.

## 4. Tick 1 observed event
* Must be an accepted `observe` candidate.
* `claim_scope` **must be** `observed`.
* Evidence list **must include** a single entry with `category: observed_world_fact`
  (or any other allowed `observed`‑related category).
* The candidate is passed to `verify_candidate_event(candidate, existing_events)` and then
  appended with `append_event(ledger_path, candidate)`.

## 5. Tick 2 dependent memory/speech event
Two conceptual lanes are allowed:
1. **Concrete speech lane** – use an existing speech mapper (`whisper` or `help`).
   * `claim_scope` **must be** `speech`.
   * Evidence **must include** `category: agent_speech`.
   * May also include a `world_event` evidence entry referencing Tick 1.
2. **Future memory lane** – a planned `memory` path.
   * `claim_scope` **must be** `memory`.
   * Evidence **must include** `category: agent_memory`.
   * No mapper currently exists; the spec only describes the intended shape.
   * May also include a `world_event` reference to Tick 1.

The Tick 2 event **does not** inherit `observed` truth; it remains a speech or memory record.

## 6. Claim‑scope preservation rules
* `observed` stays `observed` – only events generated by the `observe` mapper may use this scope.
* `speech` stays `speech` – a speech‑derived candidate never upgrades to `observed`.
* `memory` stays `memory` – a future memory‑derived candidate remains memory.
* Prior events may *influence* later events, but influence does **not** change the later event’s `claim_scope`.
* The ledger records each event separately; the `claim_scope` of each record is preserved.

## 7. Evidence category requirements
| Tick | Required `claim_scope` | Required evidence category |
|------|------------------------|---------------------------|
| 1    | `observed`             | `observed_world_fact` (or other allowed observed category) |
| 2‑a  | `speech`               | `agent_speech` |
| 2‑b  | `memory` (future)      | `agent_memory` |

An optional additional entry with `category: world_event` may be added to Tick 2 to cite the Tick 1 `event_id`.  Re‑using Tick 1’s `observed_world_fact` evidence alone does **not** satisfy the scope‑matching rule for Tick 2.

## 8. Prior‑event reference / provenance strategy
* Add a `world_event` evidence entry to the Tick 2 candidate.
* The entry’s `reference` (or `ref`) field points to the Tick 1 `event_id`.
* This provides provenance while keeping the two records distinct and preserving their individual `claim_scope`s.

## 9. Export / readback expectations
* `read_events(ledger_path)` must return two distinct records with unique `event_id`s and the correct `claim_scope`s (`observed` for Tick 1, `speech` or `memory` for Tick 2).
* `export_events(events, fmt=..., fields=..., strict=...)` must emit a representation that retains the original `claim_scope`s and does **not** promote the Tick 2 event to `observed`.
* Export formats (JSON, JSONL, etc.) must contain the `world_event` evidence linking back to Tick 1 when present.

## 10. Reject paths
1. Tick 1 `observe` candidate is rejected or never appended.
2. Tick 2 candidate lacks evidence whose category matches its `claim_scope`.
3. Tick 2 incorrectly declares `claim_scope: observed` based only on influence from Tick 1.
4. Tick 2 omits a `world_event` provenance entry when the harness intends to prove an echo.
5. Verifier reports duplicate `event_id` or contradictory evidence leading to rejection.

## 11. Tempdir‑only proof plan for the future harness
1. Create a temporary ledger directory.
2. **Tick 1** – Adam generates an `observe` candidate → `verify_candidate_event` → `append_event`.
3. **Tick 2** – Eve generates a `whisper` (or `help`) candidate that includes:
   * `claim_scope: speech`
   * `agent_speech` evidence
   * `world_event` evidence referencing Tick 1’s `event_id`
   → `verify_candidate_event` → `append_event`.
4. Call `read_events` to ensure two distinct records exist.
5. Export the ledger with `export_events` and assert that both scopes are present and unchanged.
6. Clean up the temporary directory.

This plan proves the echo contract using only existing code paths while leaving the memory lane as a documented future extension.

## 12. Open questions
* Should the concrete speech lane for Tick 2 prefer `whisper` or `help` for the first executable proof?
* When a dedicated `memory` mapper is added, what minimal signature should it expose to stay compatible with `candidate_from_*` conventions?
* How strict should the `world_event` provenance entry be (required fields, format) for future validation?
* Should export validation include both JSON and JSONL checks, or is a single format sufficient for the proof?
